# 实时计算补传数据补偿算法结论

## 结论

实时报警补传补偿不应使用当前 `uuid` 作为同一事件的唯一判断依据。当前 `uuid` 包含事件开始点时间，补传点一旦修正开始时间，就会导致 `uuid` 变化。补偿算法应把“实时已保存事件列表”和“补偿重算正确事件列表”标准化成统一事件片段，再按终端、报警类型、报警状态和时间区间重叠进行匹配。

## 推荐流程

1. 取受影响旧事件：查询所有与核心补偿区间有交集的旧事件，避免漏掉开始早于核心区间但持续覆盖核心区间的事件。
2. 重算正确事件：从原始 0200 点表中取比核心区间更宽的 warm-up / look-ahead 点序列，按 GPS 时间排序后重新跑规则。
3. 匹配旧事件和新事件：按 `terminalId`、`alarmType`、`alarmStatus` 和时间重叠匹配，选择重叠比例最高的候选。
4. 生成修正动作：将差异归为 `NOOP`、`UPDATE`、`INSERT`、`CANCEL`。
5. 专门落库：新增补偿保存方法处理修正动作，不复用追加型实时保存逻辑。

## 表结构建议

如果业务允许保留修正历史，优先增加可追溯字段：

- `correction_status`: `NORMAL` / `CORRECTED` / `CANCELED`
- `correction_source`: `REALTIME` / `COMPENSATE`
- `correction_time`
- `revision`

这样可以避免补偿任务直接删除或覆盖事件后无法追踪原因。

## 边界

这条结论来自一次算法方案记录，尚未与实际表结构、现有代码实现和业务展示规则逐项核对。落地前还需要确认是否允许物理删除事件、`event_id` 是否可更新，以及前端是否依赖旧事件 ID 展示历史。
