# 全图监控锁车流程要点

## 核心结论

海外车联网运营平台全图监控锁车，是“前端入口判断 + 权限与验证码校验 + 机构类型分流 + 命令策略下发 + 回调状态回显”的完整闭环。排查锁车问题时，不能只看最终是否下发 `LOCK`，还要确认前置 `ifLock`、机构类型、当前锁车状态、指令类型和回调状态。

## 主流程

1. 用户在运营平台全图监控选中车辆，前端查询车辆信息。
2. 前端调用 `/lockBusiness/ifLock` 判断是否可操作；`type != 0` 且具备 `1105-1` 权限才展示锁车/解锁入口。
3. `ifLock.type = 1` 展示去锁车/锁车，`ifLock.type = 2` 展示去解锁/解锁。
4. 用户确认后，前端 POST `/lockBusiness/lock`，进入 `admin-YK lock()`。
5. 后端再次执行 `ifLock` 校验，不允许则返回无权限；允许则校验短信验证码 `checkCode`。
6. 校验通过后，按机构类型组织指令：本部、当地、商户等组织 `AUTO_LOCK`；其他机构组织普通 `LOCK`。
7. `business-center /command/pushCmd` 构造 `CommandContext`，执行 `ExecuteCommand.before()`，加载车辆、终端、锁车状态，并尝试撤销未完成指令。
8. 按 `CommandEnum` 选择 [[overseas-telematics/concepts/AUTO_LOCK自动锁车策略|AUTO_LOCK自动锁车策略]] 或普通 `LockCommandStrategy`。
9. 下发结果返回后，前端按 code 展示提示；最终等待锁车结果回调，并用 `/lockBusiness/lockStatusQuery` 刷新状态。

## 状态与结果

- `200` 或 `3004`：前端提示操作成功，请稍后刷新页面查看结果。
- `50001`：前端提示错误并关闭弹窗。
- 其他失败：前端不展示成功，按现有错误处理。
- `LOCK` 成功回调：更新 `retryLockStatus = 3`，`lockStatusQuery` 显示已锁定。
- `LOCK` 失败回调：回退或更新失败状态，通常回到未锁定。
- 等待或离线：保持流程中/等待状态，`lockStatusQuery` 显示锁车中。

## 排查重点

- 入口不展示：先查 `ifLock` 的可操作性、`type` 和 `1105-1` 权限。
- 点击后直接失败：先查后端二次 `ifLock`、短信验证码 `checkCode` 和机构类型。
- 已提示成功但页面没变：先查业务中心返回 code、锁车结果回调和 `lockStatusQuery`。
- 自动锁车链路异常：重点看锁车中台当前状态，以及实际下发的是 `ENABLE`、`ACTIVE` 还是 `LOCK`。
- 与老控制器问题联动排查时，再结合 [[overseas-telematics/concepts/驾驶循环锁车时序|驾驶循环锁车时序]] 判断是否需要下一驾驶循环再次下发。
